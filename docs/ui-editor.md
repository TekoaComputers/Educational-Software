# ui_editor.js — runtime UI tweaker

A dormant script that activates only when called from the browser
console. Use it to drag/resize UI elements live, capture the deltas
as a JSON diff, and paste the diff to me — I map selectors to source
files and apply the changes.

## Usage

Navigate to the screen you want to tweak, then in the dev-tools console:

```js
loadUiEditor()        // attaches handles to every visible positioned element
                      // + shows a small control panel top-right

// (drag the red dot to move; drag the blue square at bottom-right to resize)

copyUiEditorDiff()    // JSON to clipboard
unloadUiEditor()      // remove handles + hide panel (changes preserved)

ui_editor.rescan()    // re-attach after the app rendered new content
ui_editor.reset()     // restore every changed element to its baseline
ui_editor.changes()   // inspect changes inline
```

The control panel also has buttons for each of the above.

## What gets a handle

Every element whose computed `position` is `absolute` or `fixed`, with
visible bounds (≥ 6 px). That matches the way every site positions its
controls (Kesem `.ctrl`, hemed_nivim `.ctrl`, Mikraot stage layers,
etc.).

## Diff format

```json
{
  "url": "https://…/hemed_nivim_site/#/Nivim/unit/37/connect",
  "app": "Nivim",
  "screen": "Nivim.unit.37.connect",
  "viewport": { "w": 1440, "h": 900 },
  "changes": [
    {
      "selector": "body > div.stage > div.ctrl.connect-box.kind-a:nth-of-type(3)",
      "from": { "left": 100, "top": 200, "width": 60, "height": 30 },
      "to":   { "left": 120, "top": 205, "width": 60, "height": 30 },
      "tag":  "div",
      "text": "לפעמים"
    }
  ]
}
```

Selectors are synthesized from tag + class + nth-of-type. They're
specific enough to land back on the same element when re-rendered, and
descriptive enough that I can read the source file and find the right
spot.

## Mapping back to source

I do this per-site by hand:

- `hemed_nivim_site` — positions are in `js/apps/*.js` strings like
  `b.style.cssText = "left:120px; top:205px; …"`.
- `Kesem_site` — positions live in JSON layout files under
  `data/layout/<App>/*.json` (regenerated by `tools/parse_frm.py`).
  Apply the deltas there, then rebuild the bundle.
- `Mikraot_site` — `js/renderer.js` + per-screen JS files.
- `Tirgolit_site` — `.frm`-derived inline styles in `js/*.js`.
- `makhela_site` — JS-driven; usually `data/hotspots.js` for hit zones
  and inline styles for static UI.

## Don'ts

- **Don't ship UI edits without testing** — the handle drag updates
  inline `style.left/top/width/height`, which CSS-overrides any source
  values. The changes you paste me are visual-only until I apply them
  to source and rebuild.
- **Don't expect handles to follow the app** — when the app rerenders
  a screen (e.g. unit picker → game screen), call `ui_editor.rescan()`
  to attach handles to the new DOM.
- **Don't enable on production** unless you're actively debugging —
  the script idles at <1 KB on disk + zero runtime cost until you call
  `loadUiEditor()`, but the buttons could be mis-pressed if you bring
  up the panel and walk away from the tab.
