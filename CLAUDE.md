# Tekoa Computers — educational software port

Web port of the Tekoa Computers VB6 educational suite (Hebrew). Multiple
subapps share patterns. Original VB6 source for each app lives at the
repo root under `<AppName>/` (untracked — Brahot/, EnglishA/, KolKoreA/,
etc.) and is the source of truth for porting decisions.

This file is **mirrored as `AGENTS.md`** (symlink). Edit either; both
update. Claude Code auto-loads `CLAUDE.md`; other AI tools usually look
for `AGENTS.md`.

## Build / dev loop

The Kesem suite is bundled. **Source lives in `Kesem_site/js/` and
`Kesem_site/tools/build_bundle.py`. `Kesem_site/kesem.bundle.js` is
GENERATED — never edit it directly.**

```bash
cd Kesem_site
python3 tools/build_bundle.py   # after editing js/**, tools/build_bundle.py,
                                # or data/layout/**/*.json
```

The other subsites (`Tirgolit_site/`, `hemed_nivim_site/`, `makhela_site/`)
load source JS directly — no build step.

## Tree map

- `index.html` — main catalog landing page; links to every subapp.
- `main_site_assets/feedback.js` — floating "משוב" widget; captures
  `console.*` + opens a prefilled GitHub issue. Wired into every subsite's
  `index.html`. Earlier feedback bodies live on the issues themselves.
- `Kesem_site/` — bundled SPA covering Brahot, Hagim, Yeled, Dvash,
  Heshbon, Ivrit, Shabat, Shirim, Shirim&Meshalim, KolKoreA–D, EnglishA–C.
  - `index.html` ↔ shell
  - `kesem.bundle.js` ↔ **GENERATED** — do not edit
  - `tools/build_bundle.py` ↔ bundler **plus most game-logic JS lives
    inline here** (engine has the renderer; game flow + per-game handlers
    are in this file). Edit here, then rebuild.
  - `tools/parse_frm.py` ↔ VB6 `.frm` → JSON. Property whitelist in
    `NUM_PROPS` / `STRING_PROPS`; non-whitelisted props are DROPPED. If
    you need one in the renderer, add it and re-parse.
  - `tools/parse_paths.py` ↔ binary `.MAS` / `.RAS` parsing.
  - `js/engine/frmRenderer.js` ↔ generic VB6 form renderer.
  - `js/apps/<App>.config.js` ↔ per-app screen list + image bindings.
  - `data/layout/<App>/*.json` ↔ parsed forms (regenerate via parse_frm.py).
  - `data/layout/_shared/*.json` ↔ shared `Games*.frm` layouts
    (games / games2 / games3 / games4 / games5 / misger / mashal).
  - `data/paths/<App>.json` ↔ stage data (.RAS hotspot rects, .MAS slots).
  - `data/tafrosh/<App>.json` ↔ tooltip strings.
  - `assets/<App>/menu/*.png` ↔ icons, lamps, plane sprites (from .bmp).
  - `assets/<App>/bmp/*.png` ↔ stage images.
  - `assets/<App>/wav/*.wav` ↔ audio.
- `Tirgolit_site/` — **Tirgolit / Tirgolit2** (תרגולית — math drill,
  integers + fractions). Stand-alone, source JS loaded directly. Game
  modules: `game.js` (main rooster), `game2.js`, `game3.js`, `gamewar.js`
  (WarG canvas duel), `gamekrav.js` (Krav canvas combat), `gamesketch.js`
  (Tirgolit2 freehand math grid), `vkeyboard.js` (virtual keyboard for
  RTL input), `users.js` (per-student data + bank editor), `audio.js`,
  `debug.js`. Data piped through `data.js` / `data2.js` (generated by
  `generate_data.py`). VB6 forms ported with 1:1 pixel coords from
  `Tirgolit/*.frm` (login, units, glist, game, score, krav, war).
- `hemed_nivim_site/` — **חמ"ד (Tzemed-Hemed)** and **ניבים ופתגמים**
  (idioms/proverbs). Stand-alone, single fixed 800×600 stage with
  positioned divs at twips/15 coords. Seven sub-games per app:
  `american.js` (4-choice quiz), `apple.js` (apple-picking), `connect.js`
  (text-to-image matching, fixed with real `measureText` text-width
  calculation), `hakira.js` (investigation), `haklada.js` (typing),
  `hatamaplus.js` (matching+1), `match.js` (memory pairs). Per-question
  error tracking + nikod score board built in. `tools/port_hemed_nivim.py`
  is the porting script (forms + assets). **The original `Hemed.exe` /
  `Nivim.exe` binaries are broken** (silently exit even on a complete
  install); the web port IS the canonical source — no `.frm` to refer
  back to mid-development.
- `makhela_site/` — **מקהלה (Aliza)** music-education game. Originally
  shipped as DOS; we have `makhela.jsdos` archived. **Rebuilt from
  scratch as pure JS** (no jsdos), with assets extracted via
  `tools/extract_instr_b.py`, `instr_b_explorer.py`, `sprite_explorer.py`
  and the `finish_midi_picker.html` browser tool. Data files: `songs.js`
  (per-song MIDI + lyrics), `menu_order.js`, `hotspots.js` (clickable
  regions), `midi-blob.js` + `instrument-blob.js` (binary blobs as
  base64). `vendor/picoaudio/` plays MIDI; `js/app.js` is the router,
  `js/screens.js` has every screen renderer. Sub-games: instrument
  picker (pan_inst + mud1), memory game, game-show quiz, sprite
  explorer / settings screen.
- `<AppName>/` at repo root (e.g. `Brahot/`, `EnglishA/`, `KolKoreA/`,
  `Yeled/`, …) — **untracked** dirs holding original VB6 sources: `.frm`,
  `.bas`, `.cls`, `SST.frx` (binary), `.exe`, `MENU/*.bmp`, `BMP/`, etc.
  Always read these before porting.

## Hard-won porting conventions

**Always read the original `.frm` before guessing.** The source is in
cp1255 (Hebrew Windows codepage):
```bash
tr -d '\r' < EnglishA/Sst.frm | iconv -f WINDOWS-1255 -t UTF-8 | sed -n '<range>p'
```

**No headless-Chrome testing.** The user prefers source-grounded
diagnosis to runtime probing. The feedback widget captures real-user
console logs into GitHub issues — use those for repro context.

**Coordinate system gotcha — twips/15, not /12.** VB6 stores
`Left/Top/Width/Height` in TWIPS regardless of ScaleMode. These forms use
`ScaleMode=Pixel` with `ScaleWidth ≈ ClientWidth/12` (authored at 120
DPI). But `DpiFix.bas` short-circuits when `Screen.TwipsPerPixelX = 15`
(96 DPI), so the original ran at 96 DPI. **Renderer uses
`TWIPS_PER_PX = 15` (frmRenderer.js:331).** The `px` field in layout
JSON was computed at 12 t/px — **IGNORE it**, do the math yourself.

**Z-order is inverted.** VB6 `.frm` lists controls earlier-first = drawn
on TOP. CSS stacks later-DOM = on top. Every time you set a
`backgroundColor`/`backgroundImage` on a control, double-check that it
doesn't cover something that VB6 had above it. Already-hit cases:
- `Picture1` over `Picture2` (game5 tiles ate clicks until tile z-index
  was raised)
- `Picture5` cyan-strip over `PicTime` plane (PicTime z=11 to stay above)
- `caftpla.png` appended to `PicPlace` after `lblToz` → buried the score
  markers (use `insertBefore(im, picPlace.firstChild)`)

**Transparent `VB.Label` is click-through in VB6.** In CSS divs block
clicks. `frmRenderer.js` special-cases this: any `VB.Label` that isn't a
hotspot or tooltip gets `pointer-events: none`. If a click handler stops
firing because a "transparent" label sits over it, extend that rule.

**`AutoSize = -1` PictureBox resizes to its image's natural dims at
runtime.** The `.frm` Width/Height is a design-time placeholder.
`frmRenderer.js` honors this for any control with a bound `imgSrc`.

**Per-app subs differ; do not assume one app generalises.**
Each `<App>/Sst.frm` `Lampas()` does its own thing with `btnLamp`. Each
`<App>/Games*.frm` `Init_Pic`/`InitGraph`/etc. has its own numbers.
Examples:
- `Lampas()` prints score `v` only on KolKoreA/B/C/D (with app-specific
  ForeColor + CurrentX/CurrentY).
- `Picture1.Visible = False` happens only in `Games5.frm` Act().
- `nikod.frm`'s `adv` toggle position is hand-tuned per app
  (`NIKOD_ADV_POS` table in `build_bundle.py`).

**Asset case is unstable.** Originals mix `XSST.BMP`, `tem_11.bmp`,
`Hetz5.BMP`. Windows is case-insensitive; our converter lowercases to
`.png`. If a load 404s, check casing — also the `port_app.sh` /
`sync_menu_assets.sh` scripts under `Kesem_site/tools/`.

**The Kesem bundle is concatenated, not minified.** `kesem.bundle.js` is
~3 MB of human-readable JS. Search/edit-friendly. Grep for symbol names
to find call sites quickly.

## Workflow for fixing a reported issue

1. Read the issue body — usually has URL, UA, and the feedback widget's
   captured `[kesem] ...` console log.
2. `gh issue list --repo TekoaComputers/Educational-Software --state open`
   for context. If unauthenticated, fall back to:
   ```bash
   curl -sS -H "Accept: application/vnd.github+json" \
     "https://api.github.com/repos/TekoaComputers/Educational-Software/issues?state=open"
   ```
3. Locate the app + screen + control name. Read the original `.frm`
   (`grep -n <Control> <App>/*.frm`, then `sed -n '<range>p'`).
4. Cross-check `.bas` / `.cls` modules referenced from the form
   (`DpiFix`, `Global`, `ResChange`, `FIO`, `LEV`, etc.).
5. Trace our port: most game logic is in `Kesem_site/tools/build_bundle.py`;
   renderer-level concerns are in `js/engine/frmRenderer.js`.
6. Edit the source. `python3 Kesem_site/tools/build_bundle.py` to rebuild.
7. Tell the user "ready to test." **Do NOT commit before they confirm.**
8. After confirmation, commit with `closes #N` trailer so the push
   auto-closes the issue. The Kesem bundle and its source are both
   modified; stage both.

## Maintain this file

This file is loaded into every AI session's context — that means new
insights get amplified across all future work, but it also means
**every line costs context budget**. Treat additions as a privilege, not
a habit.

**Worth adding** (propose to the user during the session, then update):
- A new VB6 quirk or runtime gotcha that wasn't in the conventions
  section and would have saved hours if you'd known it earlier.
- A new repo/subsite pattern (e.g. how a particular sub-game wires up).
- A discovered failure mode worth flagging ("if X happens, suspect Y").
- A new build / extraction / parsing tool, or a non-obvious step in an
  existing one.

**Not worth adding** (use the live-state queries or `git log` instead):
- Status snapshots (open issues, recent commits, current TODO).
- Things obvious from reading the code or asset names.
- One-off bug fixes — the commit message + `git log` already record them.
- Restatements of what's already here.

**Style for additions:**
- Terse, pointer-heavy. Link to a file path (with `:line` if useful)
  instead of pasting code.
- New rules go under "Hard-won porting conventions" or under the
  relevant subsite's paragraph in the tree map.
- If a section grows past ~15 lines, summarise inline and move detail
  to a longer doc under `docs/`.

**Process:** when you spot an insight, mention it to the user, then
either edit this file as part of the same change set or open a tiny
follow-up. Don't silently rewrite the file.
